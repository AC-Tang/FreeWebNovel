# FreeFiction（白嫖小说网）— 面试项目介绍

---

## 一、项目概述（30秒电梯演讲）

这是一个**前后端分离的全栈小说阅读平台**，用户可以免费在线阅读小说、发表评论、参与互动。项目支持小说爬虫自动化采集、JWT 双 Token 用户认证、Redis 缓存优化、AI 智能推荐等企业级功能，拥有完整的前台阅读系统和管理后台。

- **项目名称**：FreeFiction（白嫖小说网）
- **项目类型**：全栈 Web 应用
- **开发周期**：个人独立开发
- **代码规模**：后端 20+ 张数据表、80+ 个 API 接口；前端 21 条路由、30+ 个页面/组件

## 四、核心功能模块

### 4.1 用户认证系统（JWT 双 Token 机制）

这是我重点设计和实现的部分。

**双 Token 设计**：
- **Access Token**：有效期 1 天，用于日常 API 鉴权，存于前端 Cookie
- **Refresh Token**：有效期 10 天，用于无感刷新 Access Token

**刷新流程**：
1. Access Token 过期 → 后端返回 401
2. 前端 Axios 响应拦截器捕获 401，自动用 Refresh Token 请求刷新
3. 刷新成功 → 更新 Cookie 中的 Token，重放原请求
4. 刷新失败 → 清除 Token，跳转登录页

**为什么用双 Token？**
- 安全性：Access Token 短有效期降低泄露风险
- 用户体验：Refresh Token 实现无感续期，用户不用频繁登录
- Token 黑名单：Redis 维护已注销 Token 的黑名单，支持主动登出

**密码安全**：BCrypt 加密存储，即使数据库泄露也无法还原明文

### 4.2 小说自动化采集系统

这是项目最有技术含量的模块，实现了一整套自动化小说爬虫流水线。

**采集流程**：

```
管理员触发 → 获取小说信息(HTML页面) → IDM下载完整TXT → 编码转换(GBK→UTF-8)
           → 章节解析(按章节标题正则分章) → 逐章入库 → 更新统计信息(字数/章节数)
```

**技术细节**：
- **小说源**：从第三方小说站（qishuxia.com）爬取
- **信息爬取**：Jsoup 解析 HTML 页面，提取书名、作者、简介、封面、分类、章节列表
- **文件下载**：集成 IDM（Internet Download Manager）命令行进行大文件高速下载
- **编码处理**：源站使用 GBK 编码，下载后统一转换为 UTF-8
- **章节解析**：正则匹配章节标题模式（"第X章"），将完整 TXT 切分为独立章节
- **智能更新**：对比本地最新章节与远程最新章节列表，增量下载新章节
- **容错机制**：随机延迟避免反爬、失败重试、进度跟踪、异常日志

### 4.3 小说阅读系统

**阅读器功能**：
- 章节内容分页展示
- 自定义阅读设置：字体大小、背景色（日间/夜间模式）、行间距
- 章节导航：上一章/下一章、目录侧边栏
- 阅读进度记忆：自动记录阅读位置，下次打开自动跳转
- 段评功能：点击段落发表段评论（类似微信读书）
- 弹幕打卡：章节内发送弹幕式评论

### 4.4 社区互动系统

- **评论系统**：支持小说评论、章节评论、段落评论三种粒度
- **评分系统**：用户可以对小说打分（5分制），系统计算平均评分
- **书架管理**：用户自定义书架分类，支持二级分类
- **签到系统**：每日签到获取积分
- **排行榜**：多维度排行（人气榜、推荐榜、收藏榜、新书榜、完结榜、点击榜）
- **反馈/举报**：用户反馈建议 + 举报违规内容
- **通知系统**：系统消息推送

### 4.5 AI 智能助手

- 集成 DeepSeek API，提供 AI 聊天功能
- 支持流式响应（SSE），打字机效果输出
- AI 根据用户描述推荐书籍
- 支持多轮对话历史

### 4.6 管理后台

管理员功能模块：
- **仪表盘**：数据概览
- **书籍管理**：书籍增删改查、上下架
- **爬虫管理**：触发爬虫采集、批量操作
- **用户管理**：用户列表、角色管理

## 五、项目亮点（面试重点突出）

### 1. JWT 双 Token 无感刷新
不只是简单的 Token 验证，而是实现了完整的 **Access Token + Refresh Token** 机制：
- 后端 JWT Filter 校验、Redis Token 黑名单
- 前端 Axios 拦截器自动刷新、请求队列重放（多个并发 401 请求只刷新一次 Token）
- 这才是企业级的认证方案，不是 tutorial 级别的实现

### 2. 自动化小说爬虫流水线
- 从信息爬取 → 文件下载 → 编码转换 → 章节解析 → 数据入库，全链路自动化
- 集成 IDM 下载引擎处理大文件
- 增量更新机制，避免重复下载
- 容错与反爬策略

### 3. Redis 缓存优化
- 小说信息、搜索结果、排行榜等热点数据缓存
- 分级缓存过期策略（小说信息、搜索结果等不同 TTL）
- 数据更新时主动清除相关缓存

### 4. 完整的工程规范
- 统一响应格式（Result<T>）
- 全局异常处理（@RestControllerAdvice）
- ServiceResult 模式（业务层统一返回风格）
- API 文档自动生成（Knife4j）
- 前端 ErrorBoundary 组件容错
- ESLint + Prettier 代码规范

### 5. 前后端完整闭环
- 从数据库设计 → 后端 API → 前端页面 → 部署，一个人完成所有环节
- 完整的权限控制：公开页面 / 需登录 / 管理员 三级权限
- Vue Router 路由守卫 + Pinia 状态管理

## 七、面试常见问题准备

### Q1：项目中遇到的难点是什么？怎么解决的？

**爬虫编码问题**：第三方小说站使用 GBK 编码，而系统统一用 UTF-8。刚开始直接读取导致乱码。解决方式是：先用 `Files.readAllBytes()` 以二进制读取，再按 GBK 解码为字符串，最后以 UTF-8 写回文件。

**Token 刷新的并发问题**：当多个请求同时返回 401 时，如果每个请求都去刷新 Token，会导致多次刷新请求。我的解决方案是用一个 `isRefreshing` 标志 + 请求订阅者队列：第一个 401 触发刷新时，后续 401 请求先挂起等待，刷新成功后统一重放。

### Q2：为什么选择 MyBatis-Plus 而不是 JPA？

- 对 SQL 的控制力更强，复杂查询更灵活
- 自带分页插件、条件构造器，开发效率高
- 国内使用更广泛，学习资料多

### Q3：Redis 在项目中的具体用途？

- **热点数据缓存**：小说信息、搜索结果、排行榜，减少数据库压力
- **缓存失效策略**：数据更新时主动清除相关缓存 key
- **Token 黑名单**（可选实现）：用户登出后将 Token 加入 Redis 黑名单

### Q4：如何保证系统安全？

- Spring Security 统一认证授权
- BCrypt 密码加密，不存明文
- JWT 无状态认证，避免 Session 固定攻击
- CORS 跨域配置
- 全局异常处理避免敏感信息泄露

### Q5：如果重新做，会做什么改进？

- 引入 Elasticsearch 做全文搜索（已有依赖但未充分使用）
- 将爬虫模块独立为微服务，通过消息队列解耦
- 增加单元测试覆盖率
- 前端引入 TypeScript
